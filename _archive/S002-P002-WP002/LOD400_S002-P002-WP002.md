## 1. Scope Reminder

This WP implements a static API key authentication middleware for the FastAPI REST API. A single secret key is stored in the `API_KEY` environment variable. Clients pass the key in the `X-API-Key` HTTP header. The middleware validates the key using constant-time comparison and returns 401 for missing or invalid keys. Health, docs, and OpenAPI endpoints are exempt. This is a single-user, single-key model -- no rotation, no multi-key, no JWT. Those capabilities are deferred to S003.

## 2. Technical Specification

### LOD300→LOD400 Refinement Note

The parent LOD300 (v2.1.1) defines the auth interface as a FastAPI router dependency `verify_api_key(request)` attached via `Depends()` to auth-required routers. This LOD400 implements that contract exactly. The component diagram in LOD300 §2 uses "Auth Dependency" as a conceptual label for the request processing flow — the actual mechanism is router-level dependency injection, not ASGI middleware. The LOD300 §5 interface contract uses `verify_api_key(request)` (dependency signature), not the earlier `auth_middleware(request, call_next)` (middleware-stack signature).

### 2.1 Middleware Implementation

**File:** `shaked_wg_agent/api/auth.py`

**What to implement:**

1. Create a FastAPI dependency function `verify_api_key` that accepts a `Request` object. (The function name `verify_api_key` is preferred over `auth_middleware` to avoid confusion with ASGI middleware; both names are acceptable.)
2. Register it as a router-level dependency using `Depends()` injection. Create an `auth_router = APIRouter(dependencies=[Depends(verify_api_key)])` and mount all auth-required routes on it. Health/docs routes remain on a separate router (or the main app) with no auth dependency.
3. The dependency function signature:

```python
from fastapi import Depends, HTTPException, Request

async def verify_api_key(request: Request) -> None:
    """
    FastAPI dependency that validates the X-API-Key header.
    Raises HTTPException(401) if the key is missing or invalid.
    Exempt paths are handled by not attaching this dependency to exempt routers.
    """
```

4. In `api/app.py` `create_app()`: mount `auth_router` (search, listings, runs routes) and a separate `public_router` (health). FastAPI auto-serves `/docs` and `/openapi.json` on the main app, which has no auth dependency.

**Acceptance criteria:**
- [ ] AC-01: `verify_api_key` is a callable that can be used with `Depends()`.
- [ ] AC-02: Auth-required routes invoke `verify_api_key` on every request.
- [ ] AC-03: Exempt routes (`/health`, `/docs`, `/openapi.json`) never invoke `verify_api_key`.

### 2.2 API Key Validation

**What to implement:**

1. Read the expected key from `os.environ.get("API_KEY")`.
2. Read the provided key from `request.headers.get("X-API-Key")`.
3. If `API_KEY` env var is not set (is `None` or empty string), raise `HTTPException(status_code=500)` with body `{"error": {"code": "INTERNAL_ERROR", "message": "Server misconfiguration"}}`. Do NOT reveal that the API key is unconfigured.
4. If the `X-API-Key` header is missing, raise `HTTPException(status_code=401)`.
5. If the header is present, compare using `hmac.compare_digest()`:

```python
import hmac

if not hmac.compare_digest(provided_key, expected_key):
    raise HTTPException(status_code=401, ...)
```

6. Both `provided_key` and `expected_key` must be `str` types passed to `hmac.compare_digest()`. No encoding to bytes is required (Python 3.x `compare_digest` accepts `str`).
7. If comparison succeeds, the dependency returns `None` (request proceeds to route handler).

**Acceptance criteria:**
- [ ] AC-04: Comparison uses `hmac.compare_digest()` -- verified by code review or by mocking and asserting the call.
- [ ] AC-05: Invalid key returns 401 with the exact error body specified in section 5.
- [ ] AC-06: Missing key returns 401 with the exact same error body (no differentiation).
- [ ] AC-07: Missing `API_KEY` env var returns 500 on auth-required endpoints.

### 2.3 Environment Variable

**Variable:** `API_KEY`

**What to implement:**

1. The middleware reads `API_KEY` from `os.environ` on every request (not cached at import time). This allows key changes without server restart.
2. At server startup, log a WARNING if `API_KEY` is not set or is shorter than 32 characters. Use `logging.getLogger("shaked_wg_agent.api.middleware")`.
3. Do NOT refuse to start if the key is missing or short -- only warn.
4. The `/health` endpoint response includes an `auth_configured` boolean field:

```python
{
    "status": "ok",
    "version": "...",
    "auth_configured": True  # or False if API_KEY is unset/empty
}
```

**Acceptance criteria:**
- [ ] AC-08: `API_KEY` is read from `os.environ` per-request, not cached.
- [ ] AC-09: Server starts without `API_KEY` set (no crash).
- [ ] AC-10: `/health` response includes `"auth_configured": false` when `API_KEY` is unset.
- [ ] AC-11: `/health` response includes `"auth_configured": true` when `API_KEY` is set.

### 2.4 Exempt Routes

**Exempt paths (no auth required):**

| Path | Reason |
|------|--------|
| `/health` | Operational monitoring -- must be reachable by load balancers and uptime checks |
| `/docs` | Swagger UI -- useful for debugging, no sensitive data |
| `/openapi.json` | OpenAPI schema -- required by Swagger UI |

**What to implement:**

Use the router-level dependency approach: do not add `Depends(verify_api_key)` to the health/public router. FastAPI serves `/docs` and `/openapi.json` automatically on the main app, which does not have the auth dependency. Only the `auth_router` (data endpoints) gets it.

**Acceptance criteria:**
- [ ] AC-12: `GET /health` returns 200 without any `X-API-Key` header.
- [ ] AC-13: `GET /docs` returns 200 without any `X-API-Key` header.
- [ ] AC-14: `GET /openapi.json` returns 200 without any `X-API-Key` header.

### 2.5 Error Response

**What to implement:**

The 401 response MUST use the same `ErrorResponse` envelope defined in S002-P002-WP001:

```json
{
    "error": {
        "code": "UNAUTHORIZED",
        "message": "Missing or invalid API key",
        "detail": null
    }
}
```

1. The response body is identical for missing key and invalid key. No information leakage.
2. The `Content-Type` header is `application/json`.
3. No `WWW-Authenticate` header is returned (this is API key auth, not HTTP Basic/Bearer).
4. The API key value MUST NOT appear in logs, error messages, or response bodies. Log only: `"Auth failed: missing key"` or `"Auth failed: invalid key"` at DEBUG level.

**Implementation with FastAPI HTTPException:**

```python
from fastapi import HTTPException
from fastapi.responses import JSONResponse

raise HTTPException(
    status_code=401,
    detail={
        "error": {
            "code": "UNAUTHORIZED",
            "message": "Missing or invalid API key",
            "detail": None,
        }
    },
)
```

Note: FastAPI wraps `detail` in a `{"detail": ...}` envelope by default. To produce the exact response body, either:
- Use a custom exception handler that returns `JSONResponse` directly, or
- Return a `JSONResponse(status_code=401, content={...})` instead of raising `HTTPException`.

The builder must ensure the final HTTP response body matches the schema exactly. Test against the literal JSON.

**Acceptance criteria:**
- [ ] AC-15: 401 response body matches the exact JSON schema above.
- [ ] AC-16: 401 response for missing key and invalid key are byte-identical.
- [ ] AC-17: API key value never appears in server logs.

## 4. API Contract Changes

All auth-required endpoints now require the `X-API-Key` header:

| Endpoint | Method | Header Required | Behavior Without Header |
|----------|--------|-----------------|------------------------|
| `/health` | GET | No | 200 OK (exempt) |
| `/docs` | GET | No | 200 OK (exempt) |
| `/openapi.json` | GET | No | 200 OK (exempt) |
| `/search` | POST | `X-API-Key: <key>` | 401 UNAUTHORIZED |
| `/listings` | GET | `X-API-Key: <key>` | 401 UNAUTHORIZED |
| `/listings/{id}` | GET | `X-API-Key: <key>` | 401 UNAUTHORIZED |
| `/runs` | GET | `X-API-Key: <key>` | 401 UNAUTHORIZED |
| `/runs/{id}` | GET | `X-API-Key: <key>` | 401 UNAUTHORIZED |

**Header specification:**

| Header | Value | Required |
|--------|-------|----------|
| `X-API-Key` | The API key string (min 32 chars recommended) | Yes, for data endpoints |

**OpenAPI annotation:** The auth scheme should be declared in the FastAPI app so it appears in `/docs`:

```python
from fastapi.security import APIKeyHeader

api_key_header = APIKeyHeader(name="X-API-Key", auto_error=False)
```

Pass `api_key_header` as a parameter to `verify_api_key` so Swagger UI shows the "Authorize" button.

## 5. Error Handling Requirements

| Error Case | HTTP Status | Response Body | Log Message (DEBUG) |
|------------|-------------|---------------|---------------------|
| `X-API-Key` header missing | 401 | `{"error": {"code": "UNAUTHORIZED", "message": "Missing or invalid API key", "detail": null}}` | `Auth failed: missing key` |
| `X-API-Key` header present but wrong value | 401 | `{"error": {"code": "UNAUTHORIZED", "message": "Missing or invalid API key", "detail": null}}` | `Auth failed: invalid key` |
| `X-API-Key` header present, correct value | _(pass through)_ | _(no error)_ | _(no log)_ |
| `API_KEY` env var not set, auth-required endpoint | 500 | `{"error": {"code": "INTERNAL_ERROR", "message": "Server misconfiguration", "detail": null}}` | `Auth error: API_KEY env var not configured` |
| `API_KEY` env var not set, exempt endpoint | _(pass through)_ | _(no error -- exempt)_ | _(no log)_ |

**Key invariants:**
- Missing key and wrong key produce byte-identical responses. An attacker cannot distinguish the two.
- The actual key value never appears in any log line, error message, or response.
- The 500 for unconfigured `API_KEY` does not reveal the nature of the misconfiguration to the client.

## 6. Out of Scope (Explicit)

- **Key rotation** -- NOT included. Single static key. Deferred to S003.
- **Multi-key support** -- NOT included. One key for the entire API. Deferred to S003.
- **JWT authentication** -- NOT included. S003 replaces static key with per-user JWT tokens.
- **RBAC (role-based access control)** -- NOT included. S003 adds admin/agent/viewer roles.
- **Rate limiting** -- NOT included. Single-user; no abuse vector. Deferred to S003.
- **Key generation CLI** -- NOT included. Operator creates key manually and puts it in `.env`.
- **API key in query parameter** -- NOT supported. Header only. Query param `api_key` is silently ignored.
- **Per-endpoint auth levels** -- NOT included. All data endpoints have the same auth requirement.
