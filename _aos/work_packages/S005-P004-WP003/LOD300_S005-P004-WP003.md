# Keyword and Label Locale Generalization — LOD300 System Design

**work_package_id:** S005-P004-WP003
**parent_lod200:** LOD200_S005-P004-WP003.md v1.1.0

---

## 1. Executive Summary

This WP extracts hardcoded German keywords and labels from scorer, scrapers, CLI, and notifiers into a locale-keyed registry. The registry is a Python module (`shaked_wg_agent/locale.py`) — the simplest approach for a 2-locale system where keywords change only with code releases.

**Key design decision:** Python dict in a dedicated module (not JSON file, not config schema). Keywords are developer-managed, change infrequently, and benefit from type checking.

**Effort estimate:** ~3-4 hours. Extract keywords, create locale module, update consumers, add Hebrew equivalents. Medium risk — touches multiple files but each change is mechanical.

---

## 2. Design Decision: Registry Structure

### 2.1 Options Evaluated

| Option | Pros | Cons | Verdict |
|--------|------|------|---------|
| **A. JSON file** (`data/locales/de-CH.json`) | Editable without code change; familiar format | Extra file I/O; no type checking; separate from code that uses it; overkill for 2 locales | REJECTED |
| **B. Python dict in module** (`locale.py`) | Type-checked; co-located with code; zero file I/O; simple imports | Requires code release to change keywords | **SELECTED** |
| **C. Config section in city/profile JSON** | Ties keywords to geography | Pollutes city/profile schema; keywords are not geographic properties; complex to query | REJECTED |

### 2.2 Decision Rationale

- Keywords change only when a new market is added (infrequent, requires code changes anyway)
- Only 2 locales now (de-CH, he-IL); no need for runtime locale loading
- Python module enables IDE autocomplete and import-time validation
- Aligns with D1 (low business potential → avoid over-engineering)

## 5. Data Model

### 5.1 Locale dataclass (`locale.py`)

```python
@dataclass(frozen=True)
class Locale:
    """Locale-specific keywords and labels for a market."""
    
    # Language
    accept_language: str          # e.g. "de-CH,de;q=0.9,en;q=0.8"
    
    # Vegan scoring keywords (used by scorer.py)
    vegan_strong: frozenset[str]  # highest score
    vegan_partial: frozenset[str] # medium score
    vegan_weak: frozenset[str]    # low score
    vegan_no_signal: frozenset[str]  # "kein signal" equivalents
    
    # CLI status labels
    status_labels: dict[str, str]  # {"favorit": "⭐ favorit", ...}
    
    # Transit label
    transit_label: str            # "Transit lines" (generic, same for all)
    
    # Display
    currency_symbol: str          # "CHF", "₪" — for compact display
    
    # Direction (consumed by WP004 for RTL; defined here as locale property)
    direction: str                # "ltr" or "rtl"
    html_lang: str                # "de" or "he" — for <html lang="...">
```

### 5.3 Registry and accessor

```python
LOCALES: dict[str, Locale] = {
    "CH": LOCALE_CH,
    "IL": LOCALE_IL,
}

_DEFAULT_LOCALE = LOCALE_CH

def get_locale(country: str = "CH") -> Locale:
    """Return locale for country code. Falls back to Swiss German."""
    return LOCALES.get(country, _DEFAULT_LOCALE)
```

## 8. Business Rules

1. **Locale resolution:** `CityDefinition.country` → `get_locale(country)` → Locale object
2. **Fallback:** Unknown country code → Swiss German locale (safe default)
3. **Swiss scraper keywords stay German:** wgzimmer/flatfox/wg-gesucht parse German sites. Their parsing patterns are source-specific, not locale-specific.
4. **Scorer keywords are locale-specific:** Vegan signal matching must use the correct language for the listing's origin country
5. **Status labels are UI-level:** German labels for Swiss users, Hebrew labels for Israeli users. Determined by the active city's country.
6. **100% locale rule:** When an IL profile is active, the ENTIRE interface is Hebrew + RTL — all controls, labels, status badges, everything. No mixing of languages or text directions within a single report/CLI session. CH = 100% German + LTR.
7. **Region = one country:** A region/city cannot span multiple countries. Each CityDefinition belongs to exactly one country, which determines the complete locale.
8. **RTL scope boundary:** This WP provides the locale data layer (keywords, labels, language codes). The RTL rendering implementation (HTML `dir="rtl"`, bootstrap.rtl.min.css, CSS logical properties, `dir="ltr"` for price tokens) is WP004 scope. WP004 MUST follow AOS Module 11 (RTL_BIDI_STANDARD_v1.0.0.md) exactly.
