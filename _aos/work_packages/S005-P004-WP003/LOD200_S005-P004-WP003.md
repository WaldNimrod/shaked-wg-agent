# Keyword and Label Locale Generalization — LOD200 Concept

**work_package_id:** S005-P004-WP003

**priority:** LOW — German keywords work for Swiss sources; Hebrew needed only when IL scrapers exist
**approved_by:** _pending_
**approved_at:** _pending_

## 1. Problem Statement

Scoring keywords (vegan/vegetarian), scraper Accept-Language headers, CLI status badges, and email notifier text are all hardcoded in German. The scorer's vegan keyword sets (`scorer.py`, lines 21-23) contain only German terms ("vegan", "vegetarisch", "pflanzlich"). Four scrapers hardcode `Accept-Language: de-CH` in requests. CLI status badges use German labels. Email notifier uses German text.

For Israeli listings, Hebrew equivalents are needed. The current architecture has no locale-awareness — adding Hebrew requires either duplicating logic or extracting keywords into a locale-keyed configuration.

## 2. Proposed Solution (concept only)

Extract all locale-specific keywords and labels into a configuration registry keyed by locale (e.g., `de-CH`, `he-IL`). Each city definition's `country` field determines which locale to use. Scorers, scrapers, and UI components consume locale-appropriate values.

## 3. Scope

### 3.1 In scope

**3.1.1 Keyword locale registry**

- Extract German vegan keyword sets from `scorer.py` (lines 21-23) into locale-keyed config
- Extract German keywords from 4 scrapers (`wgzimmer_pw.py`, `wgzimmer.py`, `wg_gesucht.py`, `flatfox.py`)
- Add Hebrew equivalents ("טבעוני", "צמחוני", etc.)
- Design registry structure (LOD300 required — options: JSON file, Python dict, or config.py section)

**3.1.2 Accept-Language parameterization**

- Replace hardcoded `de-CH` in `BaseScraper` (`base.py`, line 20) with locale from city/source definition
- Each source definition provides its own Accept-Language header

**3.1.3 CLI label parameterization**

- Replace German status badges in `__main__.py` (lines 138-144) with locale-aware labels

**3.1.4 Email notifier text**

- Replace German text in `notifier/email_notifier.py` with locale-aware strings

### 3.2 Out of scope (explicit)
