# S002 v0.3.0 Production Deployment — LOD200 Concept

**work_package_id:** S002-OPS-WP001
**domain:** shaked-wg-agent
**track:** A — direct to LOD400 (operational, single-step deployment)
**approved_by:** _pending_
**approved_at:** _pending_

## 1. Problem Statement

S002 implementation is complete (L-GATE_V PASS, 2026-04-12) but lives only on the `dev` branch. Production still runs S001 v0.2.2. The new features — three-entity data model, REST API, API key auth, multi-channel notifications, multi-city support — are not live. Shaked has no visibility into what changed or access to the updated system.

## 2. Proposed Solution (concept only)

Execute a full deployment pipeline: push code to main, deploy to production servers via existing SFTP infrastructure, run a full Basel scan to populate fresh data, verify the published report is accessible, and send Shaked a notification summarizing what changed in S002.

## 3. Scope

### 3.1 In scope

- Git push `dev` → `main` (merge or fast-forward)
- Deploy updated code to production server (existing SFTP/cron infrastructure)
- Run full Basel scan via CLI: `python -m shaked_wg_agent run --profile default`
- Verify published HTML report is live at the public URL
- Send Shaked a deployment notification (Telegram or email) explaining S002 changes:
  - Three-entity config model (cities, profiles, sources)
  - REST API available (localhost, not publicly exposed yet)
  - API key authentication
  - Multi-channel notification framework
  - Multi-city readiness (Zurich + Bern city definitions, no profiles yet)
  - 81 tests, canonical developer docs

### 3.2 Out of scope (explicit)

- New server infrastructure or CI/CD pipeline changes
- Public API server deployment (uvicorn not exposed externally — local only)
- Activating Zurich/Bern profiles (city definitions exist but no search profiles created)
- Notification channel setup (channels list is empty in default profile — framework ready, not configured)
- S003 work

## 4. Affected Components

| Component | Nature of change |
|-----------|-----------------|
| git branches (dev → main) | merge |
| Production server files | deploy (overwrite) |
| `data/listings.json` | refresh via scan |
| `data/runs.json` | new run record |
| Published HTML report | regenerate |
| Shaked (notification recipient) | receive deployment summary |

## 5. Dependencies

- Server SSH/SFTP access (existing credentials in `.env`)
- `UPRESS_*` environment variables configured on server
- Telegram bot token or SMTP credentials for notification to Shaked
- Production server cron job points to correct code path

## 6. Risks

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|-----------|
| SFTP credentials expired | LOW | MED | Test connection before full deploy |
| Cron still runs S001 code path | LOW | MED | Verify cron target path matches deploy target |
| New config format breaks on server (missing data/cities/) | MED | HIGH | Ensure full data/ directory deployed, not just code |
| Playwright not installed on server | LOW | MED | wgzimmer scraper returns [] gracefully; flatfox works without it |
