# Facebook Email Notification Parser — Automated Passive Acquisition — LOD200 Concept

**work_package_id:** S005-P005-WP003
**domain:** shaked-wg-agent
**track:** A — pattern-following, builds on WP002's LLM parser
**priority:** MEDIUM — supplements manual input with passive automation
**approved_by:** team_190 (PASS_WITH_FINDINGS)
**approved_at:** 2026-04-15

## 1. Problem Statement

WP002 solves the Hebrew parsing problem but relies on manual copy-paste — the user must actively browse Facebook groups and capture posts. This creates a coverage gap: posts published when the user is not actively browsing are missed.

Facebook sends **email notification digests** to group members when new posts are published. These emails contain post snippets (text, author, group name, link to post) and are sent automatically without any scraping or ToS violation. By parsing these emails, we can passively discover new rental posts and feed them into the same LLM extraction pipeline from WP002.

This approach has **zero legal risk**: the user subscribed to email notifications through Facebook's official mechanism, and we are processing emails the user explicitly receives. No automated access to Facebook is involved.

## 2. Proposed Solution (concept only)

Create an **EmailFacebookScraper** — a BaseScraper subclass that reads Facebook group notification emails from a local mailbox (IMAP or exported files), extracts post snippets, feeds them through the WP002 LLM parser, and produces ScrapedListing entries.

The pipeline:
1. Connect to an email account (IMAP) or read from a local export directory
2. Filter for Facebook group notification emails (sender: `notification@facebookmail.com`, subject contains group name)
3. Parse HTML email body to extract: post text snippet, author name, group name, post URL, timestamp
4. Feed extracted text through the `parse_rental_post()` function from WP002
5. Deduplicate against existing listings and against the manual input file
6. Output ScrapedListing entries

**Coverage characteristics:**
- Facebook batches notifications — not every post triggers an email
- Facebook's algorithm selects which posts to notify about (high-engagement, recent)
- Coverage is partial (~30-60% of posts) but requires zero manual effort
- Complements WP002's manual input: manual for high-priority posts, email for passive coverage

## 3. Scope

### 3.1 In scope

**3.1.1 EmailFacebookScraper class**

- New file: `shaked_wg_agent/scrapers/facebook_email.py`
- Subclasses `BaseScraper`, implements `fetch_listings() -> list[ScrapedListing]`
- Two input modes:
  - **IMAP mode:** Connect to email server, fetch unread FB notification emails, mark as processed
  - **File mode:** Read from a local directory of exported `.eml` or `.html` email files
- Input mode configured via `search_url` field in sources.json:
  - IMAP: `imap://env/INBOX` (host, user, password are ALWAYS read from env vars — never inline in URI or config)
  - File: `data/facebook/emails/` (directory path)
- **Credential contract (canonical):** All IMAP credentials MUST be supplied via environment variables only. Inline credentials in URIs, config files, or sources.json are explicitly forbidden.
  - `SHAKED_EMAIL_HOST` — IMAP server hostname
  - `SHAKED_EMAIL_USER` — IMAP username
  - `SHAKED_EMAIL_PASS` — IMAP password
  - If any of these 3 env vars are unset when IMAP mode is configured: `fetch_listings()` returns `[]` with WARNING log

**3.1.2 Facebook notification email parsing**

- Parse HTML email body from Facebook group notifications
- Extract:
  - Post text snippet (may be truncated — Facebook emails often show first 200-300 chars)
  - Group name and group URL
  - Post URL (direct link to the Facebook post)
  - Timestamp from email headers
  - Author display name (for classification only, NOT stored in output)
- Handle different Facebook email notification formats:
  - "New post in [Group Name]" — single post notification
  - "Digest: N new posts in [Group Name]" — batch digest with multiple snippets
  - "Popular posts in [Group Name]" — algorithmic selection
- Robust HTML parsing (Facebook email HTML is complex with nested tables and tracking pixels)

**3.1.3 Integration with WP002 LLM parser**

- Reuse `parse_rental_post()` from `shaked_wg_agent/parsers/llm_listing_parser.py`
- Pass extracted text snippets through the same LLM extraction pipeline
- Handle truncated text gracefully (LLM extracts what it can; missing fields = null)
- If post text is too short for meaningful extraction (<50 chars), skip it

**3.1.4 IMAP connection management**

- Secure connection (IMAPS / STARTTLS)
- Credentials via environment variables: `SHAKED_EMAIL_HOST`, `SHAKED_EMAIL_USER`, `SHAKED_EMAIL_PASS`
- Configurable IMAP folder (default: INBOX)
- Filter criteria: sender contains "facebookmail.com", subject matches configured group names
- After processing: mark emails as read (IMAP flag) to avoid re-processing
- Connection timeout and retry logic (same pattern as BaseScraper._get retry)
- Graceful failure: IMAP not configured -> return `[]`

**3.1.5 Deduplication**

Deduplication follows a deterministic ordered pipeline. Each step is evaluated in order; a match at any step causes the listing to be skipped.

| Priority | Method | Scope | Match criteria |
|----------|--------|-------|----------------|
| 1 | Email message-ID | Within email source | Already-processed message-ID → skip entire email |
| 2 | Post URL exact match | Cross-source (email vs manual) | `raw_url` matches existing listing's `direct_url` → skip |
| 3 | Text hash | Within email batch | SHA-256 of normalized post text matches another post in same batch → first wins |
| 4 | Fuzzy listing match | Cross-source (all sources) | city + price (±10%) + rooms match existing listing → skip, optionally log as `cross_listed` |

- Track processed email message-IDs in a local state file (`data/facebook/processed_email_ids.json`) to avoid re-processing on next run
- Tie-breaker: if same listing found via both manual (WP002) and email (WP003), the manual entry takes precedence (assumed higher quality due to full text)

**3.1.6 Source registration**

- Add source entry in sources.json:
  ```json
  {
    "source_id": "facebook-email",
    "label": "Facebook Groups (email)",
    "base_url": "https://www.facebook.com",
    "scraper_class": "shaked_wg_agent.scrapers.facebook_email.EmailFacebookScraper",
    "requires_playwright": false,
    "notes": "Parses FB group notification emails. Passive acquisition. Requires IMAP config.",
    "city_params": {
      "pardes-hanna-region": {
        "search_url": "imap://env/INBOX",
        "connection_method": "email_imap",
        "enabled": true
      }
    }
  }
  ```

**3.1.7 Privacy safeguards**

- Same PII stripping as WP002: no phone numbers, no author names in output
- Email credentials stored only in environment variables, never in config files
- Processed email content is not persisted (only the extracted ScrapedListing output)

### 3.2 Out of scope (explicit)

- Email account creation or configuration (user sets up their own email + FB notifications)
- Gmail API or OAuth-based email access (IMAP is simpler and more universal)
- Push/webhook email reception (polling-based via IMAP is sufficient for 2x/day scans)
- Fetching full Facebook posts from truncated email snippets (would require FB access)
- Automated Facebook notification subscription management
- Support for non-Facebook email notifications (WhatsApp, Telegram groups)
