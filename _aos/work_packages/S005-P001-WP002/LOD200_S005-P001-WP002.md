# Yad2 Connector Architecture + Normalizer Gateway — LOD200 Concept

**work_package_id:** S005-P001-WP002
**domain:** shaked-wg-agent
**track:** B — requires LOD300 design (new integration pattern)
**approved_by:** _pending_

**depends_on:** S005-P001-WP001 (Yad2 POC must be COMPLETE first)

## 1. Problem Statement

Yad2 is behind aggressive Cloudflare anti-bot measures and is expected to change its defenses frequently. A standard `BaseScraper` subclass would tightly couple the volatile Yad2-specific scraping logic with the stable core data pipeline. Team 00 mandates architectural separation: the Yad2 connector must be an independent module with a normalizer gateway at the integration boundary.

## 2. Proposed Solution (concept only)

Create a dedicated `connectors/yad2/` subpackage that encapsulates all Yad2-specific logic (HTTP handling, Cloudflare bypass, response parsing). A normalizer gateway converts Yad2's raw data format into `ScrapedListing` objects. The gateway is the only integration point with the core system. The connector can be updated, replaced, or disabled without touching core code.

## 3. Scope

### 3.1 In scope

- Connector subpackage architecture: `shaked_wg_agent/connectors/yad2/`
- Normalizer gateway API contract (input: Yad2 raw data → output: `ScrapedListing`)
- Error handling strategy (Cloudflare blocks, rate limits, partial data)
- Registration in `sources.json` with connector-specific `scraper_class` routing
- Dynamic scraper registry in `runner.py` (replace hardcoded mapping)
- Maintenance runbook: how to update when Yad2 changes defenses
- NIS→CHF price handling at normalizer level (or currency-agnostic `ScrapedListing`)

### 3.2 Out of scope (explicit)

- Actual Yad2 scraping implementation (that depends on POC findings)
- Other Israeli source connectors
- Scoring model changes for Israeli signals
- Production deployment
