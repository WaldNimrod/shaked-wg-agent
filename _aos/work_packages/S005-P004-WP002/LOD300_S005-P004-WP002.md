# Dynamic Scraper Registry — LOD300 System Design

**work_package_id:** S005-P004-WP002
**parent_lod200:** LOD200_S005-P004-WP002.md v1.1.0

---

## 1. Executive Summary

This WP replaces the hardcoded scraper mapping in `runner.py` (lines 113-127) with a data-driven registry that resolves scraper classes from `sources.json`. It also introduces a `connector_class` field for the Yad2-style normalizer gateway and extracts the flatfox-specific verification logic from the runner into the scraper module.

**Effort estimate:** ~1-2 hours. Single file refactor (runner.py) + sources.json update. Low risk.

---

## 2. System Behavior Overview

**Current state (`runner.py:113-127`):**
```python
def _build_scraper(source, city):
    from shaked_wg_agent.scrapers.flatfox import FlatfoxScraper
    from shaked_wg_agent.scrapers.wg_gesucht import WgGesuchtScraper
    from shaked_wg_agent.scrapers.wgzimmer_pw import WgzimmerPlaywrightScraper

    mapping = {
        "wgzimmer": WgzimmerPlaywrightScraper,
        "wg-gesucht": WgGesuchtScraper,
        "flatfox": FlatfoxScraper,
    }
    cls = mapping.get(source.source_id)
    if cls is None:
        raise ValueError(...)
    return cls(...)
```

**Target state:**
```python
def _build_scraper(source, city):
    cls = _resolve_class(source.scraper_class)  # dynamic resolution
    return cls(source_id=source.source_id, search_url=source.search_url, city=city)
```

The runner no longer imports scraper classes directly or maintains a mapping. New scrapers (e.g., Yad2) register by adding an entry to `sources.json`.

## 3. Component Interactions

```
sources.json                         runner.py
├── scraper_class: "FlatfoxScraper"  ├── _resolve_class(name) → class
├── connector_class: null            ├── _build_scraper(source, city)
│                                    │   └── uses scraper_class OR connector_class
▼                                    │
config.py                            ▼
├── SourceDefinition                 BaseScraper / BaseConnector
│   ├── scraper_class: str           ├── fetch_listings() → list[ScrapedListing]
│   └── connector_class: str | None  │
│                                    ▼
▼                                    flatfox.py
ResolvedSource                       ├── FlatfoxScraper
├── scraper_class: str               ├── verify_listings(listings, city)  ← extracted from runner
└── connector_class: str | None      │   (formerly _verify_flatfox_via_api)
```

**Sequence — scraper resolution:**
```
1. Runner receives ResolvedSource with scraper_class="FlatfoxScraper"
2. _resolve_class("FlatfoxScraper") → searches known module paths → returns FlatfoxScraper class
3. Validates: issubclass(cls, BaseScraper)
4. Instantiates: cls(source_id, search_url, city)
5. Calls: scraper.fetch_listings()
```

**Sequence — connector resolution (future, Yad2):**
```
1. Runner receives ResolvedSource with connector_class="Yad2Connector"
2. _resolve_class("Yad2Connector") → returns connector class
3. Validates: issubclass(cls, BaseScraper)  (connectors implement same interface)
4. Instantiates and calls same as scraper
```

## 4. Data Model Changes

### 4.1 SourceDefinition — add `connector_class`

| Field | Type | Required | Default | Description |
|-------|------|----------|---------|-------------|
| `connector_class` | `str \| None` | no | `None` | Alternative class for normalizer gateway pattern. When set, takes precedence over `scraper_class` |

### 4.2 ResolvedSource — add `connector_class`

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `connector_class` | `str \| None` | `None` | Passed through from SourceDefinition |

### 4.3 sources.json — update `scraper_class` values

Current `scraper_class` values are short names (`"WgzimmerScraper"`, `"FlatfoxScraper"`). These must be resolvable by the registry. Two options:

**Option A — Fully-qualified names:**
```json
"scraper_class": "shaked_wg_agent.scrapers.wgzimmer_pw.WgzimmerPlaywrightScraper"
```

**Option B — Short names with known module search path:**
```json
"scraper_class": "WgzimmerPlaywrightScraper"
```
Registry searches `shaked_wg_agent.scrapers.*` modules for the class name.

**Decision: Option A (fully-qualified).** Explicit, unambiguous, no search overhead. Consistent with standard Python patterns.

## 6. Verification Hook Design

The flatfox verification is the only source-specific post-scrape hook today. Rather than designing a generic hook framework (premature), the approach is:

1. Move `_verify_flatfox_via_api` logic into `scrapers/flatfox.py` as `verify_listings()`
2. Runner calls verification based on source identity:
   ```python
   # In _verify_listings():
   for lst in listings:
       source = lst.get("source")
       if source == "flatfox":
           from shaked_wg_agent.scrapers.flatfox import verify_listings as _verify_ff
           _verify_ff(flatfox_listings, city)
           break
   # ... then HEAD-check remaining
   ```
3. When a second source needs post-scrape verification, extract to a proper hook interface (not now — YAGNI).

This is intentionally minimal. A generic `post_scrape_hook` callable registry would be over-engineering for 1 source with a custom verification step.

## 7. Interface Contracts

| Interface | Change | Contract |
|-----------|--------|----------|
| `_resolve_class(fqn: str) → type` | New | Resolves FQN string to class; raises `ImportError` or `TypeError` on failure |
| `_build_scraper(source, city) → BaseScraper` | Modified | Uses `connector_class \|\| scraper_class` from source; no hardcoded mapping |
| `SourceDefinition.connector_class` | New field | `str \| None`; when set, takes precedence over `scraper_class` |
| `flatfox.verify_listings(listings, city)` | Moved from runner | Same logic, new location |

---

## 8. Business Rules

1. **Resolution precedence:** `connector_class` (if not None) takes precedence over `scraper_class`.
2. **Validation at resolve time:** Class must exist and be a subclass of `BaseScraper`. Failure raises clear error with FQN string.
3. **No lazy loading cache:** Classes are resolved per-scan, not cached. Scan frequency is 3x/day — overhead is negligible.
4. **Backward compatibility:** sources.json short names (`"WgzimmerScraper"`) are updated to FQN in same commit. No transition period needed (single deployment).
