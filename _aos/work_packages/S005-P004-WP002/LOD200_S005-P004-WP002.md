# Dynamic Scraper Registry — LOD200 Concept

**work_package_id:** S005-P004-WP002
**domain:** shaked-wg-agent
**track:** A — single component refactor
**priority:** MEDIUM — needed for any new scraper to register without code changes
**approved_by:** _pending_
**approved_at:** _pending_

## 1. Problem Statement

The scraper runner (`runner.py`, lines 113-126) uses a hardcoded dictionary mapping source names to scraper classes. Adding a new scraper (e.g., Yad2) requires modifying `runner.py._build_scraper()` directly. The existing `sources.json` already has a `scraper_class` field, but it is not used — the mapping is duplicated in Python code.

Additionally, the flatfox-specific verification logic (`_verify_flatfox_via_api`) is embedded directly in the runner, making the runner aware of source-specific implementation details.

## 2. Proposed Solution (concept only)

Replace the hardcoded mapping with a data-driven registry that uses `scraper_class` from `sources.json` for dynamic import. Support a `connector_class` field for the Yad2-style normalizer gateway pattern. Extract source-specific hooks (flatfox verification) into source-defined callables.

## 3. Scope

### 3.1 In scope

**3.1.1 Dynamic class resolution from sources.json**

- Replace hardcoded dict in `_build_scraper()` with data-driven class resolution using `scraper_class` from `sources.json`
- Runner resolves scraper class at runtime from the `scraper_class` field — exact import mechanism deferred to LOD400
- Validate resolved class is a subclass of `BaseScraper`; clear error message on resolution failure

**3.1.2 Connector class support**

- Add optional `connector_class` field to source definition schema
- When `connector_class` is set, instantiate connector instead of scraper
- Connector implements same interface as scraper (returns `list[ScrapedListing]`)
- Enables Yad2-style normalizer gateway pattern (S005-P001-WP002)

**3.1.3 Source-specific hook extraction**

- Extract `_verify_flatfox_via_api` from runner into flatfox scraper module
- Define a mechanism for source-specific post-scrape behavior (e.g., flatfox API verification) without embedding it in the runner — exact hook contract (declaration, signature, error handling) deferred to LOD400

**3.1.4 sources.json updates**

- Ensure all existing sources have `scraper_class` populated with fully-qualified class name
- Add schema documentation for `connector_class` and `post_scrape_hook` fields

### 3.2 Out of scope (explicit)

- Implementing any new scrapers or connectors
- Changes to scraper base class interface
- Plugin discovery or entry_points mechanism (premature for current scale)
- Auto-registration from filesystem scan
