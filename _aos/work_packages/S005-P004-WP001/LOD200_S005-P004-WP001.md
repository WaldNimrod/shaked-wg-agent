## 1. Problem Statement

The codebase has Swiss Franc (CHF) hardcoded into field names and display labels across 14+ files. `ScrapedListing.price_chf`, `SearchProfile.budget_min_chf`/`budget_max_chf`, and literal "CHF" strings in HTML reports, CLI, email notifier, API schemas, and digest builder all assume a single currency. Similarly, transit labels use "Tram lines" (Basel-specific) rather than generic "Transit lines".

Israeli listings use NIS (New Israeli Shekel). No listing from Yad2 or any other Israeli source can flow through the scoring pipeline, reporting, or notification channels until these fields are currency-agnostic.

## 2. Proposed Solution (concept only)

Rename currency-specific fields to generic names and add an explicit `currency` field. Update all consumers (scorer, CLI, HTML report, notifiers, API schemas, digest builder) to use the generic field names and display the currency from the field rather than a hardcoded string. Rename transit labels to generic terms.
