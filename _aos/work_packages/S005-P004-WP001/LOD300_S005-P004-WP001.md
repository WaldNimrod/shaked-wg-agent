# Data Field Generalization (Currency + Transit Labels) — LOD300 System Design

**work_package_id:** S005-P004-WP001
**parent_lod200:** LOD200_S005-P004-WP001.md v1.1.0

---

## 1. Executive Summary

This WP generalizes CHF-specific field names and labels across 14 files so any currency can flow through the pipeline. The key design decision: **currency lives on CityDefinition** (geography determines currency). ScrapedListing gains a `currency` field defaulting to the city's currency. All consumers display the currency from data, not hardcoded strings.

**Effort estimate:** ~2-3 hours of mechanical renaming + test updates. Low risk — no behavioral changes, just field names.

---

## 2. System Behavior Overview

**Current state:**
- `ScrapedListing.price_chf: int | None` — field name assumes CHF
- `SearchProfile.budget_min_chf / budget_max_chf` — field names assume CHF
- `scorer._budget_ok()` references `profile.budget_min_chf`
- HTML report, CLI, notifiers all hardcode "CHF" string
- CLI prints "Tram lines:" as transit label

**Target state:**
- `ScrapedListing.price: int | None` + `currency: str` — generic field + explicit currency
- `SearchProfile.budget_min / budget_max` — generic field names
- `CityDefinition.currency: str = "CHF"` — source of truth for currency
- All display components read `currency` from data
- CLI prints "Transit lines:" as transit label

## 3. Component Interactions

```
CityDefinition                     ScrapedListing
├── currency: str = "CHF"          ├── price: int | None
│                                  ├── currency: str   ← set by scraper from city.currency
│                                  │
▼                                  ▼
SearchProfile                      scorer.py
├── budget_min: int                ├── _budget_ok(price, profile)  ← generic names
├── budget_max: int                │
│                                  ▼
▼                                  Consumers (html_report, CLI, notifiers, API)
                                   └── display: f"{price} {currency}"  ← from data, not hardcoded
```

**Data flow — currency resolution:**
```
1. CityDefinition loaded (country="CH" → currency="CHF", country="IL" → currency="ILS")
2. Scraper creates ScrapedListing with currency=city.currency
3. ScrapedListing.to_dict() includes "price" and "currency" fields
4. scorer reads listing["price"] and profile.budget_min (generic)
5. HTML report / CLI / notifiers display listing["currency"] dynamically
```

## 4. Data Model Changes

### 4.1 CityDefinition — add `currency` field

| Field | Type | Required | Default | Description |
|-------|------|----------|---------|-------------|
| `currency` | str | no | `"CHF"` | ISO 4217 currency code for this city's market |

Existing Basel/Zurich/Bern city JSONs: no file change needed (default "CHF" applies).
New Israeli city definitions will set `currency: "ILS"`.

### 4.2 ScrapedListing — rename + add field

| Current | New | Type | Default |
|---------|-----|------|---------|
| `price_chf` | `price` | `int \| None` | `None` |
| _(new)_ | `currency` | `str` | `"CHF"` |

`to_dict()` output changes:
- `"price_chf": value` → `"price": value, "currency": "CHF"`

### 4.3 SearchProfile — rename fields

| Current | New | Type |
|---------|-----|------|
| `budget_min_chf` | `budget_min` | `int` |
| `budget_max_chf` | `budget_max` | `int` |

### 4.4 Data file changes

**`data/profiles/default.json`:**
```
"budget_min_chf": 400  →  "budget_min": 400
"budget_max_chf": 950  →  "budget_max": 950
```

**`data/listings.json` backward compatibility:**
- Existing listings have `"price_chf"` key
- `score_listing()` reads `listing.get("price") or listing.get("price_chf")` during transition
- New listings written with `"price"` + `"currency"` keys
- One-time migration: rename field in all existing listings (can be done in same commit)

## 7. Business Rules

1. **Currency source of truth:** `CityDefinition.currency` is authoritative. ScrapedListing copies city's currency at scrape time.
2. **Backward-compat transition:** Score and display code reads `listing.get("price") or listing.get("price_chf")` until all existing listings are migrated.
3. **No currency conversion:** Each market uses its own currency. Budget comparison is within same currency (profile and listings share city → same currency).
4. **Default currency:** "CHF" for all existing Swiss cities. "ILS" for Israeli cities (future).
5. **Transit label:** CLI displays "Transit lines" (generic). HTML report uses line ID without prefix assumption.
6. **Single-market per report:** A report, table, or digest is always for one city in one country. No mixed-currency rows in the same table. Each profile selects one city → one currency → one locale.
7. **100% locale rule:** When a profile is active, the entire interface uses that locale's language and direction. CH = German + LTR. IL = Hebrew + RTL. No mixing. (RTL implementation is WP004 scope; this WP provides the data plumbing so WP004 can read `currency` from data.)
