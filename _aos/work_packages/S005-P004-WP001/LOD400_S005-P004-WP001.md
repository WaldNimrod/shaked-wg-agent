## 1. Scope Reminder

This WP generalizes CHF-specific field names and labels across 14+ files so any currency can flow through the pipeline. The key design decision: currency lives on `CityDefinition` (geography determines currency). `ScrapedListing` gains a `currency` field defaulting to the city's currency. All consumers display the currency from data, not hardcoded strings. Transit labels are generalized from "Tram" to "Transit".

## 2. Technical Specification

### 2.1 CityDefinition — Add `currency` Field

**File:** `shaked_wg_agent/config.py` (lines 40–46)

**What to implement:**

1. Add field `currency: str = "CHF"` to the `CityDefinition` dataclass, positioned after the existing `country` field.
2. In `_load_city()`, read currency from JSON with default: `currency=raw.get("currency", "CHF")`.
3. Existing city JSON files (`data/cities/basel.json`, etc.) require no change — the default `"CHF"` applies.
4. **Currency code convention:** `CityDefinition.currency` stores ISO 4217 codes. CH cities use `"CHF"`. IL cities will use `"ILS"` (the ISO 4217 code for Israeli New Shekel). The display symbol (`"₪"`) is a separate concern handled by `Locale.currency_symbol` (WP003).

**Acceptance criteria:**

- AC-01: `CityDefinition` has a field `currency` of type `str` with default value `"CHF"`.
- AC-02: `_load_city("basel")` with the existing `data/cities/basel.json` (which has no `currency` key) returns a `CityDefinition` with `currency == "CHF"`.

## 4. API Contract Changes

### 4.1 ScrapedListing Construction

```python
# BEFORE (S002):
ScrapedListing(source=..., ..., price_chf=780, ...)

# AFTER (S005-P004-WP001):
ScrapedListing(source=..., ..., price=780, currency="CHF", ...)
```

### 4.2 ScrapedListing.to_dict()

```python
# BEFORE: returns {"price_chf": 780, ...}
# AFTER:  returns {"price": 780, "currency": "CHF", ...}
```

### 4.3 _budget_ok

```python
# BEFORE (S002):
def _budget_ok(price_chf: int | None, profile: SearchProfile) -> bool:
    return profile.budget_min_chf <= price_chf <= profile.budget_max_chf

# AFTER (S005-P004-WP001):
def _budget_ok(price: int | None, profile: SearchProfile) -> bool:
    return profile.budget_min <= price <= profile.budget_max
```

### 4.4 SearchProfile Fields

```python
# BEFORE: profile.budget_min_chf, profile.budget_max_chf
# AFTER:  profile.budget_min, profile.budget_max
```

### 4.5 ListingResponse (Pydantic)

```python
# BEFORE:
class ListingResponse(BaseModel):
    price_chf: int | None = None

# AFTER:
class ListingResponse(BaseModel):
    price: int | None = None
    currency: str = "CHF"
```

## 5. Error Handling

| Error case | Where | Expected behavior |
|------------|-------|-------------------|
| City JSON has no `currency` key | `_load_city()` | Default `"CHF"` applied — no error |
| Profile JSON has neither `budget_min` nor `budget_min_chf` | `_load_profile()` | `KeyError` raised — profile is malformed |
| Listing has neither `price` nor `price_chf` | `score_listing()` | `price` evaluates to `None` → `_budget_ok` returns `True` (price unknown = not filtered) |
| `currency` field missing in listing dict | Display consumers | Fall back to `listing.get("currency", "CHF")` — no error |

## 6. Out of Scope (Explicit)

1. **Currency conversion logic** — NOT built. Each market uses its own currency. No cross-currency comparison.
2. **Mixed-currency tables or reports** — NOT supported. Each report/digest/table is for one city in one country with one currency.
3. **RTL rendering** — NOT in this WP. RTL support is S005-P004-WP004 scope.
4. **New Israeli city JSON files** — NOT created. This WP adds the `currency` field; Israeli city definitions are S005-P002-WP001 scope.
5. **Locale keyword generalization** — NOT in this WP. Vegan keywords, status labels, Accept-Language are S005-P004-WP003 scope.
6. **Listings.json bulk migration script** — NOT built as a separate tool. Backward-compat fallback handles old `price_chf` keys; new listings are written with `price`+`currency`. Existing listings are updated on next scrape cycle.
