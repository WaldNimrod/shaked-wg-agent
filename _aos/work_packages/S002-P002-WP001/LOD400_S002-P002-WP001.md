## 1. Scope Reminder

This WP wraps the existing Python agent into a RESTful API using FastAPI. The API exposes three operations over HTTP with JSON request/response: triggering scans (`POST /search`), querying listings (`GET /listings`), and retrieving run history (`GET /runs`). It does NOT replace the CLI; both interfaces coexist and share the same JSON persistence layer (`data/listings.json`, `data/runs.json`). The API is a thin routing/validation/serialization layer over the existing `runner.run_scan()` and `persistence.load_listings()` / `persistence.load_runs()` functions. No business logic is duplicated.

### 2.2 FastAPI Application Setup (`app.py`)

**What to implement:**

The file exports a single factory function `create_app() -> FastAPI`.

```python
from fastapi import FastAPI
from fastapi.middleware.cors import CORSMiddleware
from contextlib import asynccontextmanager
import os, uuid
from datetime import datetime, UTC

from starlette.requests import Request
from starlette.responses import JSONResponse

from shaked_wg_agent.api.routes import router
from shaked_wg_agent.api.schemas import ErrorResponse, ErrorDetail


@asynccontextmanager
async def lifespan(app: FastAPI):
    # Startup: nothing required for S002 (no DB connections)
    yield
    # Shutdown: nothing required


def create_app() -> FastAPI:
    app = FastAPI(
        title="Shaked WG Agent API",
        version="0.3.0",
        lifespan=lifespan,
    )

    # --- CORS ---
    origins_raw = os.environ.get("API_CORS_ORIGINS", "")
    if origins_raw:
        origins = [o.strip() for o in origins_raw.split(",") if o.strip()]
        app.add_middleware(
            CORSMiddleware,
            allow_origins=origins,
            allow_methods=["*"],
            allow_headers=["*"],
        )

    # --- Request ID middleware ---
    @app.middleware("http")
    async def add_request_id(request: Request, call_next):
        request.state.request_id = f"req-{uuid.uuid4().hex[:8]}"
        response = await call_next(request)
        response.headers["X-Request-ID"] = request.state.request_id
        return response

    # --- Global exception handler ---
    @app.exception_handler(Exception)
    async def unhandled_exception_handler(request: Request, exc: Exception):
        request_id = getattr(request.state, "request_id", "unknown")
        body = ErrorResponse(
            error=ErrorDetail(
                code="INTERNAL_ERROR",
                message="An unexpected error occurred.",
                detail=str(exc) if os.environ.get("API_DEBUG") else None,
            )
        )
        return JSONResponse(
            status_code=500,
            content=body.model_dump(),
            headers={"X-Request-ID": request_id},
        )

    # --- Mount routes ---
    app.include_router(router)

    return app
```

**Key decisions:**

- CORS origins are read from `API_CORS_ORIGINS` env var (comma-separated). If unset or empty, no CORS middleware is added.
- `request_id` is generated as `req-{8 hex chars}` via `uuid.uuid4()`, stored on `request.state`, and returned in `X-Request-ID` response header.
- The global exception handler catches all unhandled exceptions and returns a 500 with `ErrorResponse` envelope. In debug mode (`API_DEBUG=1`), the `detail` field includes the exception string.
- Auth middleware is NOT implemented in this WP. It is deferred to S002-P002-WP002. All routes are currently unprotected.
- Server entry point: `uvicorn shaked_wg_agent.api.app:create_app --factory --port 8000`.

**Acceptance criteria:**
- [ ] AC-03: `uvicorn shaked_wg_agent.api.app:create_app --factory` starts without error.
- [ ] AC-04: Every response includes an `X-Request-ID` header with format `req-[8 hex]`.
- [ ] AC-05: When `API_CORS_ORIGINS=http://localhost:3000`, CORS headers are present on responses to requests with matching `Origin`.
- [ ] AC-06: When `API_CORS_ORIGINS` is unset, no CORS headers appear.
- [ ] AC-07: An unhandled exception in a route returns 500 with `ErrorResponse` JSON envelope.

### 2.3 Pydantic Request/Response Schemas (`schemas.py`)

**What to implement:**

All models use Pydantic v2 (`BaseModel`). Field serialization uses `model_dump(mode="json")`.

```python
from __future__ import annotations
from pydantic import BaseModel, Field
from typing import Generic, TypeVar
from datetime import datetime

T = TypeVar("T")


# --- Request Models ---

class SearchRequest(BaseModel):
    """POST /search body."""
    profile_id: str | None = Field(
        default=None,
        description="Search profile to scan (preferred parameter).",
    )
    city_id: str | None = Field(
        default=None,
        description="DEPRECATED. Alias for profile_id — resolved to the first "
                    "profile targeting this city.",
        json_schema_extra={"deprecated": True},
    )


class ListingsQuery(BaseModel):
    """GET /listings query parameters."""
    profile_id: str | None = None
    city_id: str | None = None
    min_score: int = Field(default=0, ge=0, le=100)
    status: str | None = None
    source: str | None = None
    limit: int = Field(default=50, ge=1, le=200)
    offset: int = Field(default=0, ge=0)


class RunsQuery(BaseModel):
    """GET /runs query parameters."""
    profile_id: str | None = None
    city_id: str | None = None
    limit: int = Field(default=20, ge=1, le=100)
    offset: int = Field(default=0, ge=0)


# --- Response Models ---

class ResponseMeta(BaseModel):
    timestamp: str
    request_id: str


class PaginatedMeta(ResponseMeta):
    total_count: int
    offset: int
    limit: int


class ResponseEnvelope(BaseModel, Generic[T]):
    data: T
    meta: ResponseMeta


class PaginatedResponse(BaseModel, Generic[T]):
    data: list[T]
    meta: PaginatedMeta


class ListingResponse(BaseModel):
    """Single listing — all fields from listings.json schema."""
    listing_id: str
    source: str
    title: str = ""
    price_chf: int | None = None
    available_from: str | None = None
    location_text: str = ""
    district: str = ""
    transit_match_lines: list[str] = Field(default_factory=list)
    roommate_signal: str = ""
    vegan_signal: str = ""
    summary: str = ""
    direct_url: str = ""
    url_status: str = ""
    relevance_score: int = 0
    status: str = "neu"
    note: str = ""
    tags: list[str] = Field(default_factory=list)
    first_seen_at: str = ""
    last_seen_at: str = ""
    verified_active: bool = False
    last_verified_at: str | None = None
    source_listing_id: str | None = None

    model_config = {"extra": "ignore"}


class RunResponse(BaseModel):
    """Single run record."""
    run_id: str
    run_timestamp: str
    triggered_by: str = "manual"
    sources_scanned: int = 0
    results_scanned: int = 0
    new_results: int = 0
    updated_results: int = 0
    stale_removed: int = 0
    duration_seconds: int = 0
    errors: list[str] = Field(default_factory=list)
    report_url: str | None = None
    operator_notes: str = ""

    model_config = {"extra": "ignore"}


class ErrorDetail(BaseModel):
    code: str
    message: str
    detail: str | None = None


class ErrorResponse(BaseModel):
    error: ErrorDetail
```

**Key decisions:**

- `SearchRequest.profile_id` is the preferred parameter. `city_id` is a deprecated alias. Resolution logic lives in `deps.py` (see 2.5), not in the schema.
- `ListingResponse.transit_match_lines` is the S002 field name. In S001, this field was called `tram_match_lines`. The rename is performed in S002-P001-WP001 (scorer update). The API schema uses the new name. Any legacy listings in `listings.json` with the old field name will have the field silently dropped by `extra="ignore"` — acceptable because existing listings are re-scored on next scan.
- `ListingResponse` and `RunResponse` use `model_config = {"extra": "ignore"}` so that any unexpected fields in the raw JSON dicts are silently dropped rather than raising validation errors. This prevents breakage if new fields are added to the persistence layer before the API schema is updated.
- `ResponseEnvelope` and `PaginatedResponse` are generic wrappers. In route handlers, they are constructed directly (not used as generic types at runtime) because Pydantic v2 generic serialization works best with concrete construction.
- `limit` fields have `le` constraints matching the maximums from LOD300: 200 for listings, 100 for runs. Values exceeding the max are rejected with 422 by Pydantic.

**Acceptance criteria:**
- [ ] AC-08: `SearchRequest(profile_id="default")` validates successfully.
- [ ] AC-09: `SearchRequest(city_id="basel")` validates successfully (deprecated alias accepted).
- [ ] AC-10: `ListingsQuery(limit=300)` raises `ValidationError` (exceeds `le=200`).
- [ ] AC-11: `ListingResponse` can be constructed from any existing `listings.json` entry without error.
- [ ] AC-12: `RunResponse` can be constructed from any existing `runs.json` entry without error.
- [ ] AC-13: `ErrorResponse` round-trips through `model_dump()` correctly.

#### 2.4.2 `POST /search`

```python
@router.post("/search")
async def search(body: SearchRequest, request: Request):
    # 1. Resolve profile_id (see deps.py resolve_profile_id)
    profile_id = resolve_profile_id(body.profile_id, body.city_id)

    # 2. Load config using resolved profile_id
    cfg = load_config(profile_id)

    # 3. Execute scan synchronously using the profile-aware config
    run_record = run_scan(cfg=cfg)
    run_record["triggered_by"] = "api"

    # 4. Build response
    return ResponseEnvelope(
        data=RunResponse(**run_record),
        meta=ResponseMeta(
            timestamp=datetime.now(UTC).isoformat(timespec="seconds"),
            request_id=request.state.request_id,
        ),
    ).model_dump()
```

**Implementation notes:**
- The resolved `profile_id` is passed to `load_config(profile_id)` which performs the full resolution chain (agent meta → profile → city → source intersection) as defined in S002-P001-WP001 §2.3. If `profile_id` is `None` (neither `profile_id` nor `city_id` provided), `load_config(None)` falls back to `agent.json → default_profile_id`.
- `run_scan(cfg=cfg)` receives the profile-aware `ProjectConfig`, ensuring the scan uses the correct city, sources, and scoring parameters for the resolved profile.
- `run_scan()` is called synchronously. The scan takes 10-30 seconds. The client must set an appropriate HTTP timeout.
- After `run_scan()` returns, override `triggered_by` from `"manual"` to `"api"`. This is a pragmatic patch until `runner.run_scan()` accepts a `triggered_by` parameter (planned in S002-P001-WP001 profile_id refactor).
- If `run_scan()` raises an exception, the global exception handler in `app.py` catches it and returns 500.

#### 2.4.3 `GET /listings`

```python
@router.get("/listings")
async def list_listings(request: Request, params: ListingsQuery = Depends()):
    # 1. Require at least city_id or profile_id
    profile_id = resolve_profile_id(params.profile_id, params.city_id)
    if profile_id is None and params.city_id is None:
        raise HTTPException(
            status_code=400,
            detail=ErrorResponse(
                error=ErrorDetail(
                    code="BAD_REQUEST",
                    message="Either profile_id or city_id is required.",
                )
            ).model_dump(),
        )

    # 2. Load all listings
    all_listings = load_listings()

    # 3. Resolve profile to city_id for filtering
    #    When profile_id is provided, load the profile to get its city_id
    effective_city_id = params.city_id
    if profile_id and not effective_city_id:
        try:
            cfg = load_config(profile_id)
            effective_city_id = cfg.city.city_id
        except (FileNotFoundError, ValueError):
            pass  # Unknown profile — skip city filter

    # 4. Filter
    filtered = all_listings
    if effective_city_id:
        filtered = [l for l in filtered if l.get("city_id") == effective_city_id]
    if profile_id:
        filtered = [l for l in filtered if l.get("profile_id") == profile_id]
    if params.min_score > 0:
        filtered = [l for l in filtered if l.get("relevance_score", 0) >= params.min_score]
    if params.status:
        filtered = [l for l in filtered if l.get("status") == params.status]
    if params.source:
        filtered = [l for l in filtered if l.get("source") == params.source]

    # 4. Sort by relevance_score descending
    filtered.sort(key=lambda l: l.get("relevance_score", 0), reverse=True)

    # 5. Paginate
    total_count = len(filtered)
    page = filtered[params.offset : params.offset + params.limit]

    # 6. Serialize
    items = [ListingResponse(**l) for l in page]
    return PaginatedResponse(
        data=items,
        meta=PaginatedMeta(
            timestamp=datetime.now(UTC).isoformat(timespec="seconds"),
            request_id=request.state.request_id,
            total_count=total_count,
            offset=params.offset,
            limit=params.limit,
        ),
    ).model_dump()
```

**Implementation notes:**
- When `profile_id` is provided without `city_id`, the profile's `city_id` is resolved via `load_config(profile_id)` and used for the city filter. The `profile_id` filter also applies directly against the listing's `profile_id` field (added by WP001 §2.5 item 13).
- Filtering is done in Python (no database). Acceptable for < 10,000 listings.
- When `offset` exceeds `total_count`, returns empty `data: []` with correct `total_count`. This is NOT an error.
- In S002, `city_id` is always `"basel"` and `profile_id` is always `"default"`.

### 2.5 Dependencies (`deps.py`)

**What to implement:**

`resolve_profile_id(profile_id: str | None, city_id: str | None) -> str | None`

A plain function (not a FastAPI `Depends()` injectable) that resolves the effective profile_id:
- If `profile_id` is provided, return it directly (preferred parameter).
- If only `city_id` is provided, scan `data/profiles/*.json` for the first profile where `city_id` matches. Return that profile's `profile_id`. If no match, return `None`.
- If neither is provided, return `None` (caller decides whether to error).

**This resolution logic is identical to the CLI's `--city` resolution in S002-P001-WP001 §2.4** — both scan `data/profiles/` directory. This ensures consistent behavior between CLI and API.

**Key decisions:**

- `resolve_profile_id` is called explicitly in each route handler (not injected via `Depends()`) because it requires the parsed request body/query params which vary per endpoint.
- In S002 (single profile "default" targeting "basel"), the directory scan is trivially fast. In S003, this will be replaced with a database lookup.
- If `city_id` is provided but no profile targets it, return `None` and let the caller decide the error response.

**Acceptance criteria:**
- [ ] AC-25: `resolve_profile_id("default", None)` returns `"default"`.
- [ ] AC-26: `resolve_profile_id(None, "basel")` returns `"default"` (scans profiles directory, finds default.json with `city_id="basel"`).
- [ ] AC-27: `resolve_profile_id(None, None)` returns `None`.
- [ ] AC-28: `resolve_profile_id("custom", "basel")` returns `"custom"` (profile_id takes precedence).
- [ ] AC-29: `resolve_profile_id(None, "nonexistent_city")` returns `None` (no profile targets this city).

## 5. Error Handling

| Error Case | HTTP Status | Code | Message | When |
|------------|-------------|------|---------|------|
| Pydantic validation failure (body or query) | 422 | `VALIDATION_ERROR` | FastAPI auto-generated detail | Invalid types, out-of-range `limit`, malformed JSON body |
| Missing required filter | 400 | `BAD_REQUEST` | `"Either profile_id or city_id is required."` | `GET /listings` with neither `city_id` nor `profile_id` |
| Resource not found | 404 | `NOT_FOUND` | `"Listing '{id}' not found."` / `"Run '{id}' not found."` | `GET /listings/{id}` or `GET /runs/{id}` with unknown ID |
| Unhandled exception | 500 | `INTERNAL_ERROR` | `"An unexpected error occurred."` | Any uncaught exception in route handlers (e.g., `run_scan()` crash, file I/O error) |
| Scan partial failure | 200 | — | — | Some scrapers fail during `POST /search`; `errors` list in `RunResponse` is non-empty but scan completes. This is NOT an HTTP error. |

**Notes:**
- 401 UNAUTHORIZED is NOT implemented in this WP. It will be added by S002-P002-WP002 (auth middleware).
- 422 responses use FastAPI's built-in Pydantic error formatting (`{"detail": [{"loc": [...], "msg": "...", "type": "..."}]}`). This does NOT use the `ErrorResponse` envelope. This is intentional to preserve FastAPI's standard validation error contract.
- The global exception handler in `app.py` catches `Exception` (not `HTTPException`) so that FastAPI's built-in 404/422 handling is preserved.
