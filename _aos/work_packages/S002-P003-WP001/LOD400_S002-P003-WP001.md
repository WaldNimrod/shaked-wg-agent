## 1. Scope Reminder

This work package implements multi-channel notification digests for the Shaked WG Basel agent. After each scan that discovers new listings, the system sends a digest summary via one or more user-selected channels: Email (SMTP), Telegram (Bot API), Discord (Webhook), ntfy (HTTPS POST), and generic Webhook (POST JSON). Notifications are opt-in, per-profile, fire-and-forget, and never block the scan. The implementation adds a `notifier/` package with a base interface and five channel modules, a digest builder, an orchestrator, and a runner integration hook.

### 2.2 Notifier Base Interface

**File:** `shaked_wg_agent/notifier/base.py`

**What to implement:**

```python
from abc import ABC, abstractmethod
from dataclasses import dataclass
from typing import Any

@dataclass
class ChannelResult:
    """Result of a single channel send attempt."""
    channel_type: str       # "email" | "telegram" | "discord" | "ntfy" | "webhook"
    label: str | None       # user-defined label from ChannelConfig, or None
    success: bool
    error: str | None       # None on success; error message on failure

class BaseNotifier(ABC):
    """Abstract base for all notification channels."""

    @abstractmethod
    def send(self, digest_payload: dict) -> bool:
        """Send the digest via this channel.

        Args:
            digest_payload: The DigestPayload dict (see section 2.8).

        Returns:
            True if the message was delivered successfully, False otherwise.

        Raises:
            Nothing — all exceptions must be caught internally and logged.
        """
        ...

    @abstractmethod
    def format_message(self, digest_payload: dict) -> Any:
        """Format the digest payload into the channel-specific message format.

        Args:
            digest_payload: The DigestPayload dict.

        Returns:
            str for plain-text channels (email HTML string, telegram markdown, ntfy text).
            dict for structured channels (discord JSON, webhook JSON).
        """
        ...
```

1. `BaseNotifier` is an ABC. Both `send()` and `format_message()` are abstract.
2. Every concrete notifier stores its `ChannelConfig` dict in `self.config` (passed via `__init__`).
3. `send()` must never raise. All exceptions are caught, logged via `logging.getLogger("shaked_wg_agent.notifier")`, and cause `send()` to return `False`.

**Acceptance criteria:**
- [ ] AC-03: Instantiating `BaseNotifier` directly raises `TypeError`.
- [ ] AC-04: A subclass that implements both methods can be instantiated.

### 2.3 Email Notifier

**File:** `shaked_wg_agent/notifier/email_notifier.py`

**What to implement:**

`EmailNotifier` extends `BaseNotifier`. Reads SMTP configuration from environment variables.

**Environment variables (read at `__init__` time):**

| Variable | Required | Default | Notes |
|----------|----------|---------|-------|
| `SMTP_HOST` | yes | — | SMTP server hostname |
| `SMTP_PORT` | no | `587` | SMTP server port |
| `SMTP_USER` | yes | — | SMTP username |
| `SMTP_PASS` | yes | — | SMTP password / app password |
| `SMTP_FROM` | yes | — | Sender address |

If any required env var is missing, set `self._available = False` and log a warning. Do not raise.

**`format_message(digest_payload) -> str`:** Returns an HTML string (inline CSS) containing:
- Header with profile_name and city_name
- Summary line: `"{total_new} neue Angebote gefunden am {scan_timestamp as DD.MM.YYYY HH:MM}"`
- Per-listing card with: title (linked to direct_url), price (`"CHF {price_chf}/Mt."` or `"Preis nicht angegeben"`), district, score (color-coded: green >= 70, orange >= 40, red < 40), vegan signal (if present), transit lines (if present)
- Footer: `"Generiert von Shaked WG Agent"`

**`send(digest_payload) -> bool`:**
- If not available, log warning and return `False`
- Send HTML email via SMTP with STARTTLS
- Subject format: `"[Shaked WG] {profile_name}: {total_new} neue Angebote"`
- Recipients from `channel_config["recipients"]`
- Return `True` on success, `False` on any exception (caught and logged)

**Acceptance criteria:**
- [ ] AC-05: `format_message()` returns valid HTML containing all listing titles from the payload.
- [ ] AC-06: `send()` calls `smtplib.SMTP`, `starttls()`, `login()`, `sendmail()` in order (verified via mock).
- [ ] AC-07: Subject line matches format `"[Shaked WG] {profile_name}: {total_new} neue Angebote"`.
- [ ] AC-08: Missing `SMTP_HOST` env var causes `send()` to return `False` with logged warning, no exception.

### 2.5 Discord Notifier

**File:** `shaked_wg_agent/notifier/discord_notifier.py`

**What to implement:**

`DiscordNotifier` extends `BaseNotifier`. Uses `webhook_url` from channel config. No environment variables needed.

- Headers: `Content-Type: application/json`
- Timeout: 10 seconds
- Success: HTTP 200 or 204 (Discord returns 204 on webhook success)
- Return `False` on any exception (caught and logged)

**Acceptance criteria:**
- [ ] AC-13: Embeds are color-coded: green for score >= 70, orange for >= 40, red for < 40.

### 2.8 Digest Builder

**File:** `shaked_wg_agent/notifier/digest_builder.py`

**What to implement:**

`build_digest_payload(profile, city, run_record, new_listings) -> dict`

Builds the DigestPayload dict consumed by all channel notifiers.

**Processing steps:**

1. **Filter:** Remove listings where `relevance_score < min_score_threshold` (from `profile["notifications"]["min_score_threshold"]`, default `0`).
2. **Sort:** Sort remaining listings by `relevance_score` descending.
3. **Truncate:** Take top N where N = `profile["notifications"]["digest_max_listings"]` (default `5`).
4. **Map to ListingSummary:** Extract per listing: `title`, `price_chf` (int or null), `district`, `relevance_score`, `direct_url`, `vegan_signal` (default `""`), `transit_match_lines` (default `[]`).
5. **Build payload:**
       "city_id": city["city_id"],
       "city_name": city["city_name"],
       "run_id": run_record["run_id"],
       "scan_timestamp": run_record["timestamp"],  # ISO 8601
       "total_new": len(new_listings),              # total before filtering
       "listings": listings_summaries,               # top-N after filter+sort
   }
   ```
6. Note: `total_new` reflects the total count of new listings found by the scan (before score filtering), so the user knows the full picture even when the digest shows only top-N above threshold.

**Acceptance criteria:**
- [ ] AC-25: Listings with score below `min_score_threshold` are excluded from `listings` but included in `total_new`.
- [ ] AC-26: `listings` is sorted by `relevance_score` descending.
- [ ] AC-27: `listings` contains at most `digest_max_listings` entries.
- [ ] AC-28: Default `digest_max_listings` is 5 when not specified in profile.
- [ ] AC-29: Default `min_score_threshold` is 0 when not specified in profile.

### 2.9 Orchestrator

**File:** `shaked_wg_agent/notifier/orchestrator.py`

**What to implement:**

`notify_digest(profile, city, run_record, new_listings) -> dict | None`

Main entry point. Never raises. Returns `NotificationResult` dict or `None` if notifications are not configured.

Maintains a channel registry mapping type strings to notifier classes.

**Orchestration logic:**

1. Read `profile["notifications"]`. If `None` or `channels` is empty, return `None`.
2. Filter channels: keep only entries where `enabled is True`.
3. If no enabled channels remain, return `None`.
4. Validate channel count: if > 5, log error and truncate to first 5.
5. Build digest payload once (shared across channels).
6. For each enabled channel (sequentially, not parallel):
   - Look up type in registry. Unknown type → log warning, record failure, continue.
   - Instantiate notifier with channel config.
   - Call `send()`. On success → record success.
   - On failure → classify error (transient vs permanent):
     - **Transient** (timeout, connection error, HTTP 429/5xx, SMTP connect/disconnect): retry once after 5-second delay.
     - **Permanent** (HTTP 400/401/403/404, SMTP auth error, config error): no retry.
   - Record final result per channel.
7. Return `NotificationResult` dict (see §3.3).

**Error classification:**

| Classification | Error types |
|---------------|-------------|
| Transient | `TimeoutError`, `ConnectionError`, `URLError` (connection-level), HTTP {429, 500, 502, 503, 504}, `SMTPConnectError`, `SMTPServerDisconnected` |
| Permanent | HTTP {400, 401, 403, 404}, `ValueError`, `KeyError`, `SMTPAuthenticationError` |

Each notifier exposes `self.last_error: str` and `self.last_error_transient: bool` after a failed `send()` for the orchestrator to read.

**Acceptance criteria:**
- [ ] AC-30: `notify_digest()` returns `None` when `profile["notifications"]` is `None`.
- [ ] AC-31: `notify_digest()` returns `None` when all channels have `enabled: false`.
- [ ] AC-32: Each channel is attempted independently; failure of channel N does not skip channel N+1.
- [ ] AC-33: Transient failure triggers exactly one retry after 5-second delay.
- [ ] AC-34: Permanent failure (HTTP 400/401/403) is not retried.
- [ ] AC-35: Unknown channel type logged as warning, result has `success: false`.
- [ ] AC-36: `total_sent + total_failed` equals number of enabled channels processed.
- [ ] AC-37: Maximum 5 channels enforced; excess channels truncated with error log.

### 2.10 Runner Integration

**File:** `shaked_wg_agent/runner.py` (existing file — modify)

**What to implement:**

At the end of `run_scan()`, after all scrapers have run and results are persisted, add the notification hook:

1. If `new_results > 0`, call `notify_digest(profile, city, run_record, new_listings)` and store the return value in `run_record["notification_sent"]`.
2. If `new_results == 0`, set `run_record["notification_sent"] = None`.
3. The import of `notify_digest` is deferred (inside the conditional block) to avoid circular imports and keep the notifier optional.
4. `notify_digest()` never raises, so no try/except wrapper is needed in the runner.
5. The `notification_sent` field is persisted as part of the run record written to `data/runs.json`.

**Acceptance criteria:**
- [ ] AC-38: `notify_digest()` is called exactly once when `new_results > 0` and profile has notifications configured.
- [ ] AC-39: `notify_digest()` is NOT called when `new_results == 0`.
- [ ] AC-40: `run_record["notification_sent"]` is set to the return value of `notify_digest()`.
- [ ] AC-41: `run_record["notification_sent"]` is `null` when `new_results == 0`.

## 5. Error Handling Requirements

### 5.1 Per-Channel Error Cases

| Channel | Error case | Classification | Expected behavior |
|---------|-----------|----------------|-------------------|
| **All** | Unknown channel type | Permanent | Skip, log warning, result `success: false` |
| **All** | Timeout (10s) | Transient | Retry once after 5s |
| **All** | `ConnectionError` / DNS failure | Transient | Retry once after 5s |
| **All** | HTTP 429 (rate limited) | Transient | Retry once after 5s |
| **All** | HTTP 500/502/503/504 | Transient | Retry once after 5s |
| **All** | HTTP 400 (bad request) | Permanent | No retry; log error with response body |
| **All** | HTTP 401/403 (auth failure) | Permanent | No retry; log error |
| **All** | HTTP 404 (not found) | Permanent | No retry; log error |
| **Email** | `SMTPAuthenticationError` | Permanent | No retry; log credentials warning |
| **Email** | `SMTPConnectError` | Transient | Retry once after 5s |
| **Email** | `SMTPServerDisconnected` | Transient | Retry once after 5s |
| **Email** | Missing SMTP env vars | Permanent | Skip channel, log warning at init |
| **Telegram** | Missing `TELEGRAM_BOT_TOKEN` | Permanent | Skip channel, log warning at init |
| **Discord** | HTTP 204 response | Success | 204 is normal for Discord webhooks |
| **ntfy** | Self-hosted server unreachable | Transient | Retry once after 5s |

### 5.2 Retry Logic

- Maximum 1 retry per channel per scan.
- Delay between attempts: 5 seconds (`time.sleep(5)`).
- Only transient failures are retried. Permanent failures fail immediately.
- The retry uses the same notifier instance and same digest payload.
- If both attempts fail, the error from the second attempt is recorded in the result.

### 5.3 Global Error Boundary

- `notify_digest()` wraps the entire orchestration in a top-level `try/except Exception`. If an unexpected error escapes channel-level handling (e.g., a bug in the orchestrator itself), it is caught, logged as `logger.exception(...)`, and the function returns:
  ```python
  {"channels": [], "total_sent": 0, "total_failed": 0}
  ```
  This ensures the runner never crashes due to a notification bug.

## 6. Out of Scope (Explicit)

- **Quiet hours / scheduling** — NOT included in this WP. Scans are manually triggered or cron-scheduled externally.
- **Slack** — Deferred. Requires OAuth app registration and workspace installation.
- **Matrix** — Deferred. Requires homeserver integration.
- **Web Push** — Deferred. Requires service worker and push subscription management.
- **WhatsApp** — Excluded. No free API available.
- **Signal** — Excluded. No stable public API.
- **SMS** — Excluded. Paid service required.
- **Notification history UI** — Not in scope. Results stored in run record for CLI/log inspection only.
- **Per-listing notifications** — Not in scope. Digest only (one message per scan per channel).
- **Async / parallel channel sending** — Not in scope. Channels are sent sequentially. Acceptable for 5 channels max.
- **Rate limiting across scans** — Not in scope. Each scan triggers independently.
- **Notification templates / customization** — Not in scope. Formats are hardcoded per channel.
- **Attachment support** — Not in scope. No PDFs or images in notifications.
