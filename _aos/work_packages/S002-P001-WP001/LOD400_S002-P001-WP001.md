# City-agnostic Config Schema + Scraper Interface — LOD400 Executable Specification

**work_package_id:** S002-P001-WP001
**parent_lod300:** LOD300_S002-P001-WP001.md v2.0.0

---

## 1. Scope Reminder

This work package refactors the monolithic `AgentConfig` / flat `data/config.json` configuration layer into a three-entity model separating geographic city definitions (`CityDefinition`), a global source platform registry (`SourceDefinition` with per-city `CitySourceParams`), and user search profiles (`SearchProfile`). It updates the CLI to accept a `--profile` flag, modifies `BaseScraper.__init__` to receive `CityDefinition`, and changes `score_listing` to accept `SearchProfile` instead of `AgentConfig`. The file layout moves to `data/cities/{city_id}.json`, `data/sources.json` (restructured with `city_params`), `data/profiles/{profile_id}.json`, and `data/agent.json`. A `load_config(profile_id)` function performs the full resolution chain: agent metadata, profile, city, source intersection, returning a unified `ProjectConfig`.

### 2.3 Config Resolution

**File:** `shaked_wg_agent/config.py`

Replace the existing `load_config() -> ProjectConfig` with the new signature and resolution chain.

**What to implement:**

1. Change signature: `load_config(profile_id: str | None = None) -> ProjectConfig`.
2. Resolution chain (order matters):
   - a. Call `_load_agent_meta()` to get `AgentMeta`.
   - b. If `profile_id is None`, use `agent_meta.default_profile_id`.
   - c. Call `_load_profile(profile_id)` to get `SearchProfile`.
   - d. Call `_load_city(profile.city_id)` to get `CityDefinition`.
   - e. Call `_load_sources()` to get `list[SourceDefinition]`.
   - f. Build source registry dict: `{sd.source_id: sd for sd in source_defs}`.
   - g. Determine enabled set: if `profile.enabled_sources` is non-empty, use it; else use `city.available_sources`.
   - h. Intersect enabled set with `city.available_sources`.
   - i. For each `source_id` in intersection:
     - Look up `SourceDefinition` in registry. If missing, log warning `"Source '{source_id}' in city/profile but not in sources.json — skipped"` and skip.
     - Look up `city_params[city.city_id]` on the `SourceDefinition`. If missing, log warning `"Source '{source_id}' has no params for city '{city_id}' — skipped"` and skip.
     - If `city_source_params.enabled is False`, skip.
     - Build `ResolvedSource` from `SourceDefinition` + `CitySourceParams`. Set `priority` from the source's position in `city.available_sources` (0-indexed).
   - j. If intersection yields zero active sources after filtering, raise `ValueError("No active sources after resolving profile '{profile_id}' with city '{city_id}'")`.
   - k. Return `ProjectConfig(agent=agent_meta, profile=profile, city=city, sources=resolved_sources)`.
3. Use `import logging; logger = logging.getLogger(__name__)` for all warnings — not `print()`.

**Acceptance criteria:**

- AC-21: `load_config("default")` with valid files returns `ProjectConfig` where `config.profile.profile_id == "default"`, `config.city.city_id == "basel"`, and `len(config.sources) >= 1`.
- AC-22: `load_config(None)` reads `default_profile_id` from `agent.json` and behaves identically to `load_config("default")` when `default_profile_id == "default"`.
- AC-23: `load_config("nonexistent")` raises `FileNotFoundError`.
- AC-24: When `profile.enabled_sources = []`, all `city.available_sources` are used.
- AC-25: When `profile.enabled_sources = ["flatfox"]` and `city.available_sources = ["flatfox", "wgzimmer"]`, only `flatfox` is resolved.
- AC-26: When `profile.enabled_sources = ["flatfox", "bogus"]` and `city.available_sources = ["flatfox"]`, `bogus` is logged as warning and skipped; `flatfox` resolves normally.
- AC-27: When intersection yields zero active sources, `ValueError` is raised with descriptive message.
- AC-28: Priority is assigned by position in `city.available_sources` (first = 0, second = 1, etc.).

## 6. Out of Scope

1. **Auto-migration of `data/config.json`** — manual migration script is a separate LOD500 deliverable. The old file is not read, not deleted, not validated.
2. **Adding new cities beyond Basel** — S002-P001-WP002 handles Zurich and Bern city definitions. This WP creates the schema and `data/cities/basel.json` only.
3. **New scoring dimensions** — `smoking_policy`, `rental_duration`, `custom_tags` scoring logic is NOT added in this WP. Only the data model carries these fields. Scoring WP is separate.
4. **Notification delivery** — S002-P003-WP001 implements the notification send logic. This WP defines `NotificationConfig`/`ChannelConfig` data model only.
5. **PostgreSQL migration** — S003 concern. All persistence remains JSON file-based.
6. **Multi-user isolation** — S003 concern. No `owner_user_id` fields added yet.
7. **Source scraper implementation for new cities** — only schema and interface changes. Actual scraper logic for Zurich/Bern URLs is out of scope.
8. **HTML report publisher changes** — `_publish()` in `runner.py` continues to use `cfg.agent.profile_name` (now accessed as `cfg.profile.profile_name`) and `cfg.agent.project_end` (now accessed as `cfg.agent.project_end`). Minimal adapter changes only.
